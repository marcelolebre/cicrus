Reading a system schematic

A schematic is a contract between the system and you. Once you can read the layers, the traces, and the flow direction, you can debug any service in the stack without ever logging in.

14 min read 3 exercises 1 lab Level Intermediate Last revised 2026-05-22

Every operational system you'll ever inherit is a wiring diagram in disguise. Learn to read the diagram and the system explains itself.

Section 01The shape of a schematic

A Cicrus schematic is read top to bottom. Each row is one abstraction layer — ingest at the top, storage at the bottom, application logic in the middle. Cards in a row are siblings of the same kind: three workers, four queues, two databases. The eye should never need to scan sideways for meaning.1

Press ? on any schematic page to show the legend, or L to toggle live data-flow animation on the traces.

Note

The rule is not "five rows maximum" — it's "fits on one screen without scrolling." On a typical 1080p display, that means about five rows. If your system needs more, split into two diagrams at the most natural seam.

What counts as a layer

A row is a layer if and only if every card in it has the same kind of responsibility. "Workers" is a layer; "things that happen to be running right now" is not. The discipline of choosing the right rows is half the work of drawing a useful schematic — and reading one quickly.

Layer noun
One horizontal row in a schematic. Contains cards of one uniform kind. Typically: ingest, queue, worker, storage, output.
Trace noun
A line connecting two cards. Encodes a real data flow at runtime — never a conceptual relationship.
Card noun
A single system component. Has a name, a state, a few metrics, and zero decoration.
Flow verb · also noun
The direction data moves along a trace. Always rendered top-down via dashed animation on the line itself, never via moving particles.

Section 02Layers and traces

Look at any real schematic for three seconds and you'll see two things: rows (the layers) and lines (the traces). Everything else is noise. The figure below is annotated to show what you're looking at.

Element What it encodes Visual cue Avg. count
Row One abstraction layer Horizontal band of cards 5
Card One component Bordered rectangle, name + state 3–12
Trace A data flow Line between two cards 8–24
Dot Liveness signal 1px breathing pulse on the card 1 per card
Glyph "Thinking" object state Animated pixel sphere 0–1 per screen
Tip

When debugging an unfamiliar service, find its row first — that tells you what kind of thing it is. Then trace upward to its inputs and downward to its dependencies. Never start by reading code.

Section 03Reading flow direction

Flow on a Cicrus schematic moves downward by default. Read the rows in vertical order, the same way you read a page. Bidirectional or upward flows are explicit exceptions — they're rare, and they're marked.

"A good diagram answers a question. A great diagram makes you ask the right one."
Mira Chen
Operator Handbook · §4

Find the top-most card

The system's true entry point. Usually an API gateway, a queue consumer, or a scheduled job runner. If there are multiple, mentally pick the one your request would touch first.

Trace one path downward

Follow a single trace from top card to bottom. Don't try to absorb every line at once — pick one and finish it. Repeat for the next path.

Note every fork

A card that fans out to multiple downstream cards is a decision point. List them in the margin as you go — you'll need this for diagnosis later.

Look for the missing card

An experienced reader spots what isn't drawn. If you see a worker layer but no queue feeding it, there's a queue somewhere — it just wasn't worth drawing.

Warning

Never trust an unlabeled trace. If a line between two cards has no label and you don't know what flows on it, the schematic is incomplete. File an issue against the system owner before relying on the diagram for diagnosis.

Do

Read rows top to bottom. One trace at a time. Pick a single request path and finish it before starting another.

Don't

Don't try to memorize the whole diagram before reading a trace. Don't scan sideways for "where this fits." The rows already told you.

Section 04A worked example

Below is the actual schematic for the icarus.payroll.sync service. We'll trace a single payroll run from API call to disk in four moves. Watch the sparkline on each card — that tells you the load profile of the last 30 minutes.

icarus/payroll/sync.ex 
1# entry point for the worker
2defmodule Icarus.Payroll.Sync do
3  use Icarus.Worker, queue: "payroll.sync"
4
5  def perform(%{"entity_id" => id, "cycle" => cycle}) do
6    {:ok, runs} = Payroll.load_runs(id, cycle)
7    runs
8    |> Enum.map(&FX.normalize/1)
9    |> Ledger.commit!()
10  end
11end
Example

When the payroll cycle for entity_id: "uk-ltd" enters this worker, the trace lights up green for ~210ms, then handed off to Ledger downstream. Watch for that handoff in the schematic — if the green stalls on this card, the issue is in the FX normalization step at line 8.

Lab · Trace a request ~6 min · self-checked

The system above received a request for entity_id: "br-ltda" at 14:02:17. Trace the request from the API gateway to the database. List every card it passes through, in order. Use the schematic above to verify.

Your answer should be a sequence of 4–6 component names, separated by arrows.

→

The full path is:

api.gateway → router.entity → payroll.sync → ledger.commit → store.primary

Five cards across three layers. The router.entity step is easy to miss if you only scan the topmost and bottommost rows — but it's where BR-specific FX normalization is applied.

Section 05Check your reading

Question · 1 of 1

You see a Cicrus schematic with four rows and a single trace that goes up from row 3 to row 1. What does that almost certainly indicate?

  • A The diagram is wrong — flow only goes downward.
  • B An explicit feedback path — a callback, an ack, or a push notification back to the entry point.
  • C The author drew the rows in the wrong order.
  • D A bug — the data is flowing in the wrong direction in production.
Select one answer.
Reflect

Think of a system you maintain. If you drew its schematic, what would the bottom row hold?

Section 06Summary

In summary
Four moves
Lesson 03 / 09
14 min · complete
  • Rows are the system's nouns. Every row in a schematic is one kind of thing — ingest, queue, worker, store. If a card doesn't share its row's responsibility, it's drawn in the wrong row.
  • Traces are the verbs. A line between two cards encodes a real runtime flow — never a conceptual relationship, never a wish. If you can't say what runs along it, the trace is a bug in the diagram.
  • One path at a time. Don't try to absorb the whole diagram. Pick the topmost card, follow one trace downward to the bottom, name every fork on the way. Then do it again with a different starting point.
  • The best readers see what isn't drawn. A worker layer with no queue above it tells you a queue exists somewhere — just not on this canvas. Mature operators read the gaps as carefully as the lines.
